## 描述

在实验室的运行、维护、软硬件开发、专业学术研究过程中，会产生大量且涉及多方面的知识、经验、技巧等（以下统称为“知识”）。本知识库用于将这些知识进行集中管理、维护和查阅。

- **知识库组织形式：** 将结构化的知识文档（`.md`文件）进行集中管理（位于`docs`文件夹）。文件夹内以 *分类* 且 *多级* 的目录架构储存知识文档。该知识库提供可视化网页作为接口，用于知识文档/文章的查找、阅读、下载。知识库具有可维护的能力，可进行知识文档的修改、增减、勘误、补充等。

- **知识库网页主要架构：** 知识库采用单一页面框架，其中包括 - 左边栏 & 主窗口。左边栏的内容由上到下分别为 - 知识库logo/标题、搜索栏、（多级）目录索引。主窗口用于显示主页和知识文档正文。主页是初次进入该知识库时显示的页面，其中显示知识库标题、简介、主要内容索引、开发/维护/联系信息等内容（可以参考示例`Temp_Examples\预览-实验室知识库.html`）。通过左边栏索引、搜索、或主页索引，主窗口切换进入具体知识文档正文页，该页面与结构化的知识文档（`.md`文件）连接并将其可视化。在正文页面右侧位置，还提供当前页面的辅助目录导航和下载按钮。下载按钮可以将当前知识文档保存/下载到本地。

- **知识库网页风格设计：** 以科学研究、AI、深度学习、现代化为美术风格关键词。在满足基本功能要求的前提下，适当减少其他花哨显示内容（比如华丽的动画、动态效果等）。页面、字体、文字颜色必须保证页面内容的高可读性。其他细节包括：左边栏索引的各级子目录可折叠；左边栏可向左收缩隐藏和调出。

- **部署：** 该网页可以在本地部署（需依托本地conda环境`mydocs`），也可以托管到GitHub平台并随时随地打开，可供团队小伙伴们进行知识的共享。

**实际改进方案清单 [2026-04-21 (v2)]：**

- **方案1 - 统一本地与部署环境版本（优先级：高；实施成本：低；状态：已完成）**
  问题是什么：当前本地开发依赖由 `environment.yml` 固定版本，但 GitHub Actions 部署流程直接安装最新版 `mkdocs-material` 和 `mkdocs-minify-plugin`，这会导致本地和线上环境不一致。
  为什么值得做：文档站点非常依赖主题和插件行为，一旦上游版本变化，就可能出现“本地正常、线上异常”或“本地样式正常、线上样式漂移”的问题。这类问题通常不一定直接报错，反而更难排查。
  在当前项目中的对应位置：`environment.yml`、`.github/workflows/deploy.yml`。
  建议怎么推进：让 GitHub Actions 与本地使用同一组版本来源。最简单的做法是把工作流中的依赖安装改为读取固定版本，至少确保 `mkdocs`、`mkdocs-material`、`mkdocs-minify-plugin`、`pymdown-extensions` 与本地一致。
  预期收益：减少“今天能用、下次部署突然坏掉”的不确定性，让构建结果更可复现。

- **方案2 - 建立公开文档与维护文档的边界（优先级：高；实施成本：中）**
  问题是什么：当前 `hooks/copy_sources.py` 会把 `docs/` 目录下所有 Markdown 复制到 `site/_sources/`。这意味着只要文件放在 `docs/` 下，即使没有进入导航，也可能随站点一起被发布。
  为什么值得做：现在只是有几篇维护文档没有进导航，但以后如果把草稿、内部说明、维护备忘也放进 `docs/`，就存在被一并公开的风险。这个问题通常不是“当前马上出事”，而是“以后很容易踩坑”。
  在当前项目中的对应位置：`hooks/copy_sources.py`、`docs/` 目录结构、`mkdocs.yml` 的导航配置。
  建议怎么推进：建立清晰边界，例如“公开内容只放 `docs/`”，“维护说明放仓库根目录或单独目录”；或者进一步修改 hook，只复制实际公开页面，不复制所有 Markdown。
  预期收益：避免误发布内部内容，也让文档结构更清楚，维护者更容易判断某篇内容是否面向外部。

- **方案3 - 整理 overrides 层，降低前端定制维护成本（优先级：中；实施成本：中）**
  问题是什么：当前前端定制主要集中在 `overrides/extra.css` 和 `overrides/extra.js`。随着功能增加，这两处文件会逐渐变成“所有样式和交互问题的集中堆叠点”。
  为什么值得做：这种状态在项目早期很常见，也不是错误；但如果不尽早整理，后续每修一个搜索框、侧栏、暗色模式的小问题，都可能牵动别处。项目会越来越依赖“作者记忆”而不是“代码结构”。
  在当前项目中的对应位置：`overrides/extra.css`、`overrides/extra.js`、`overrides/main.html`。
  建议怎么推进：先做轻量整理，不需要大重构。比如把 CSS 按“头部 / 搜索 / 侧栏 / 暗色主题 / 响应式”分段，把 JS 按“导航折叠 / 侧栏切换 / 搜索行为”拆成更清楚的小块，并把注释重点写在“为什么这样改”而不是“代码表面做了什么”。
  预期收益：后续查问题更快，交接更容易，也能减少未来改动时的误伤范围。

- **方案4 - 为关键交互建立最小回归检查清单（优先级：中；实施成本：低）**
  问题是什么：像搜索框这种问题，往往不是构建失败，而是交互细节不符合日常使用习惯。仅靠“代码看起来没问题”并不能保证实际体验正常。
  为什么值得做：文档站点的很多问题只有手动点击、切换、缩放后才能发现。相比引入完整自动化 UI 测试，先做一份最小回归检查清单，成本更低、收益更直接。
  在当前项目中的对应位置：搜索框、侧栏折叠、调色板切换、移动端/窄屏布局等关键交互。
  建议怎么推进：先写成一份手工检查清单。比如每次改搜索或头部后，检查“点击搜索框是否激活、输入是否有结果、`Esc` 是否关闭、点击空白处是否关闭、窄屏下是否仍正常、明暗主题下是否可见”。后续如果项目继续扩大，再考虑补自动化测试。
  预期收益：把“靠记忆检查”变成“按清单检查”，显著降低回归问题漏检概率。

- **方案5 - 清理导航遗漏页面并明确其状态（优先级：中；实施成本：低）**
  问题是什么：`mkdocs build --strict` 已提示 `docs/维护指南.md` 和 `docs/维护指南-附录-mkdocsYML.md` 存在，但未进入 `nav`。
  为什么值得做：这类问题不会立刻阻塞构建，但会让后续维护者分不清“这是故意隐藏”还是“配置漏了”。时间一久，这种不确定状态会增加维护成本。
  在当前项目中的对应位置：`mkdocs.yml` 的 `nav` 段，以及 `docs/维护指南.md`、`docs/维护指南-附录-mkdocsYML.md`。
  建议怎么推进：二选一处理即可。若它们面向读者，就加入导航；若它们只是维护文档，就移出 `docs/` 或至少在目录策略上明确为“内部文件不进 docs”。
  预期收益：消除构建告警，提升文档组织清晰度，也和方案2形成配套。

**开发日志 [2026-04-18 (v5)]：**

- **问题1修复 - 调色板按钮明亮模式可见性：** Material 主题默认 header 为深色背景，其 `.md-header__button` 图标为白色。由于本项目顶栏为浅色背景，导致图标与背景融为一体。修复方案：为 `.lab-header__palette .md-header__button` 显式设置 `color: var(--md-default-fg-color--light) !important`（灰色），hover 时变为主题色。修改文件：`overrides/extra.css`。
- **问题2修复 - 搜索框居中对齐：** 根因为顶栏 flex 布局中左右宽度不对称导致搜索框偏右。初次修复采用 `flex: 0 1 36rem` 方案，但引发搜索功能失效（见下方"重大修复"）。最终方案：将顶栏 `.md-header__inner` 从 Flexbox 改为 **CSS Grid 三栏布局** (`grid-template-columns: 1fr minmax(0, 36rem) 1fr`)，左右两栏等宽自动平衡，中间栏固定最大宽度，实现搜索框真正居中。修改文件：`overrides/main.html`（添加 `.lab-header__left` 包裹容器）、`overrides/extra.css`。
- **问题3修复 - 点击外部/Escape关闭搜索框：** Material 主题搜索通过 `#__search` checkbox 控制开关状态。添加 `initSearchDismiss()` 函数，包含三部分：(1) 聚焦搜索输入框时自动勾选 `#__search` 以激活搜索结果面板；(2) 监听 `mousedown` 事件（比 `click` 更可靠），当搜索激活且点击目标不在 `.md-search` 容器内时，取消勾选并清空输入框；(3) 监听 `keydown` 事件，Escape 键关闭搜索并清空输入。修改文件：`overrides/extra.js`。
- **重大修复 - 搜索功能失效：** 初次问题2修复后，搜索框样式无变化且搜索功能失效。根因分析：Material 主题在移动端视口（<960px）通过 CSS 将 `.md-search__inner` 设为 `height: 0; opacity: 0; width: 0; overflow: hidden`，而桌面端（>=960px）使用 `float: right; width: 11.7rem` 配合 checkbox 展开机制。自定义 flex 修改破坏了此行为链。修复方案：(1) CSS 中使用 `!important` 强制覆盖 `.md-search__inner` 的 width/height/opacity/overflow/position/transform/float，使搜索栏始终可见；(2) 添加 `.md-search__output` 绝对定位使搜索结果正确下拉展示；(3) 添加 `.md-search__overlay` 样式，搜索激活时显示半透明遮罩层。修改文件：`overrides/extra.css`、`overrides/extra.js`。
